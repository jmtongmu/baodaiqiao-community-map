# 宝带桥 3d城市版：Web 3D 引擎可行性记录

## 目标

在现有社区志地图版之外，建立一个独立的 3D 游戏引擎展示路线：不再以二维地图控件为核心，而是把高频地名、里程碑事件、时间轴和社区空间关系映射到可旋转、可点击、可动画化的 3D 城市沙盘中。

## 候选引擎

| 方案 | 适配度 | 适合原因 | 风险 |
| --- | --- | --- | --- |
| Babylon.js | 高 | 更接近 Web 游戏引擎，内置场景、相机、材质、光照、拾取、动画；适合快速搭建 3D 城市沙盘与事件交互。 | 地理 GIS 能力需要自己处理，真实地形/3D Tiles 不是强项。 |
| Three.js | 中高 | 生态广、体积相对可控，适合自定义 3D 可视化。 | 更像图形库，游戏化工程能力需要自己组织。 |
| PlayCanvas | 中 | Web 游戏引擎属性强，编辑器友好。 | 项目本地静态化和数据流水线接入成本更高。 |
| CesiumJS | 中 | 真实地理、地球、3D Tiles 和坐标体系很强。 | 更像地理空间平台，不适合做轻量游戏化城市叙事原型。 |

## 当前选择

第一版采用 **Babylon.js**。

理由：

1. 本项目当前最需要验证的是“3D 游戏引擎是否能承载社区志叙事”，而不是高精度地球浏览。
2. 现有数据已经是点、面、频次和时间轴，适合先转换为局部 3D 坐标，再用网格对象表达。
3. Babylon.js 的拾取、材质、发光、动画、相机控制足够支撑 20 秒里程碑可视化的第一阶段。

建筑层采用 **Simple 3D Buildings 思路的程序化生成方案**：先在道路、水系和地块之间自动生成现代社区建筑，再根据位置生成住宅、商业、学校、社区服务、滨水建筑等类型。这个方案不依赖卫星底图，也不等待 OSM 建筑数据完整度；后续如果从 QGIS 或 OSM 导入真实建筑轮廓，可直接替换当前自动生成的 footprint。

## 第一版验证范围

- 从 `mention_columns.geojson` 读取高频地名。
- 从 `milestones.geojson` 读取带地点的里程碑事件。
- 经纬度转为局部 3D 坐标。
- 生成式航拍城市底纹作为地面基板，程序化道路、水系和建筑块作为可控叠加层。
- 程序化现代建筑层，支持单独开关，包含窗格材质、屋顶、阳台线、商业玻璃幕墙和低层公共建筑。
- 地名提及频次映射为柱体高度。
- 里程碑事件映射为发光光柱。
- 时间轴按年份控制里程碑显隐。
- 点击对象时显示统计原因、里程碑意义和志书摘录。

## 下一步

1. 加入真实社区边界、水系、道路的 3D 底座。
2. 增加 20 秒镜头关键帧系统。
3. 增加“里程碑模式 / 高频地名模式 / 漫游模式”三种观看方式。
4. 研究是否把 QGIS 导出的建筑或地块面转成 glTF，进一步游戏引擎化。
